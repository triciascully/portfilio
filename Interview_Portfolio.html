Recommender · Built & Productionized

Content Recommendation Engine

Gannett · co-occurrence / item-item collaborative filtering

The goal
Surface the next piece of content a reader is most likely to engage with, which deepens sessions and drives subscriptions.
What we built
A content-recommendation module from scratch on a co-occurrence matrix, an item-item collaborative-filtering method that learns "what content gets consumed together," then ranks recommendations from those associations.
Why this approach
Co-occurrence is transparent, fast, and cold-start-friendly for a content catalog, with no heavy training loop, and it explains why something was recommended.
To production
We shipped it as a deployed module, not a notebook demo. And the core move, turning co-occurrence into a ranked list, which is exactly the shape of a search-ranking or client–provider matching problem.
Item-item co-occurrence matrix
Darker cells = content consumed together more often. Each row's strongest off-diagonal cells become that item's recommendations.
Technical deep-dive: math, SQL & Pythonrepresentative
The math

Co-occurrence C(i,j) = number of sessions containing both items i and j. Raw counts over-reward popular items, so we normalize, e.g. cosine: sim(i,j) = C(i,j) / √(C(i,i)·C(j,j)), or lift: C(i,j)·N / (C(i)·C(j)). Recommendations for i = the top-k items by sim, excluding i.

SQL: build the co-occurrence counts
WITH session_items AS (              -- one row per (session, item)
  SELECT session_id, content_id
  FROM page_views
  GROUP BY session_id, content_id
)
SELECT a.content_id AS item_i,
       b.content_id AS item_j,
       COUNT(*)     AS cooccur
FROM session_items a
JOIN session_items b
  ON a.session_id = b.session_id
 AND a.content_id < b.content_id     -- unordered pairs, drop self-pairs
GROUP BY item_i, item_j
HAVING COUNT(*) >= 5;                 -- prune noisy one-offs
Python: normalize & rank
import numpy as np
from scipy.sparse import csr_matrix

# C = sparse item x item co-occurrence (symmetric)
diag = np.sqrt(C.diagonal())
sim  = C.multiply(1/diag[:, None]).multiply(1/diag[None, :])   # cosine-normalize

def recommend(i, k=10):
    row = np.asarray(sim.getrow(i).todense()).ravel()
    row[i] = 0                          # never recommend the item itself
    return np.argsort(-row)[:k]         # top-k most-associated items
Why it's advanced / how it's tested

Sparse matrix math at catalog scale, a deliberate normalization choice to fight popularity bias, and a cold-start fallback to a popularity prior for new items. Evaluated offline with Recall@K / NDCG, then validated online with a CTR/engagement A/B test, the same ranking shape as a search or client–provider matching problem.

Adaptive Experimentation · Production

Multi-Armed Bandit → Live Allocation API

Gannett · Thompson Sampling · Google Cloud Function

The problem
Fixed-split A/B tests keep sending half the traffic to a variant we already suspect is losing. That opportunity cost is regret.
What we did
Compared ε-greedy, UCB, and Thompson Sampling; chose Thompson Sampling and deployed it pulling live GA4 data from BigQuery, serving allocations through a Dockerized Cloud Function.
Why this model
The true conversion rate of each variant is unknown, so we model each one as a probability distribution (a Beta distribution) that sharpens with every conversion or miss. To allocate the next users, the algorithm draws a random sample from each variant's distribution and sends traffic to whichever sampled highest. Variants that are still uncertain keep getting chances (exploration); proven winners get the bulk of traffic (exploitation), and it self-corrects automatically as data arrives, with no hand-tuned "explore X% of the time" knob like ε-greedy. "Beta-Binomial" just means the math updates instantly in closed form.
How it was measured & guard-railed
We tracked cumulative regret, meaning how many conversions the bandit gave up versus always picking the true best variant. But a bandit can over-react to a lucky early streak, so we never let it run unchecked: we still ran a real significance test (two-proportion z-test at 95%), confirmed there was enough traffic to detect a real difference (power analysis), and reported uncertainty ranges (credible intervals). So every shift in allocation was backed by evidence, not early noise.

Each draw picks the arm with the higher sampled rate. As evidence accumulates, B's posterior separates and wins more draws → more traffic.

Note the shape: early on the posteriors are wide and overlapping, so allocation swings hard from update to update. As data accumulates the swings dampen and converge on the winning split (~80/20). It hones in, rather than ramping in a straight line.
Technical deep-dive: math, SQL & Pythonreal code
The math

Model each variant's true conversion rate θ as Beta(α, β). The Beta is conjugate to the Bernoulli, so after s conversions and f misses the posterior is just Beta(1+s, 1+f), a closed-form update with no MCMC. Each round, one sample is drawn from each arm's posterior, routing the next users to argmaxₖ θ̃ₖ. Regret = Σ (θ* − θ_chosen).

SQL: pull impressions & conversions per variant (GA4 / BigQuery)
SELECT d190_page_ab_variant AS variant,
       COUNT(DISTINCT user_id) AS impressions,
       COUNT(DISTINCT IF(first_transaction_flag = 1
             AND d001_event_type LIKE '%product-purchase%',
             user_id, NULL)) AS conversions
FROM `gannett-bi-reporting...ga4_events_extracted_transformed`
WHERE d190_page_ab_variant IN ('0066_LPRB_A', '0066_LPRB_B')
  AND DATE(TIMESTAMP_MICROS(event_timestamp)) BETWEEN @start AND @end
GROUP BY variant;
Python: posterior sampling → traffic allocation

Run order: the SQL above yields (conversions, impressions) per variant → load into arms → sample each posterior → allocate traffic by win-share.

import numpy as np
N = 10_000
# one posterior per arm: Beta(1 + conversions, 1 + non-conversions)
samples = {v: np.random.beta(s + 1, f + 1, N) for v, (s, f) in arms.items()}

wins = np.zeros(len(arms))
for i in range(N):                      # Monte-Carlo over the draws
    draws = [samples[v][i] for v in arms]
    wins[np.argmax(draws)] += 1         # arm that "wins" this draw
alloc = {v: round(100 * wins[k] / N) for k, v in enumerate(arms)}  # % traffic
Python: the guardrails (statsmodels)
from statsmodels.stats.proportion import proportions_ztest, proportion_confint
from statsmodels.stats.power import NormalIndPower

stat, p = proportions_ztest(conversions, impressions)      # is the lift real?
NormalIndPower().solve_power(effect_size=None, nobs1=n,     # did we have power?
                             alpha=0.05, power=0.8, alternative='two-sided')
ci = proportion_confint(conv_b, imp_b, alpha=0.05, method='normal')  # uncertainty
Why it's advanced / how it's tested

Conjugate Bayesian updating (closed-form, cheap to run every cycle), Monte-Carlo allocation, an impression→conversion delay window, and deployment as a Dockerized Cloud Function API. Tested by tracking cumulative regret against the always-best arm, fenced by the z-test / power / credible-interval guardrails so allocation never chases early noise.

Validity QA · "The Broken Test"

Catching an Invalid Test

Gannett · EDM-1257 · sample-composition confound

The trap
A landing-page redesign read as an 18% CVR win, but the funnel was off: far more new-page users hit account creation, even though final starts matched.
What we found
The test fed a much higher share of already-registered users to the old page (≈32% vs ≈6%). Registered users skip account creation, distorting the step-to-step comparison.
Why it matters
A sample-composition / Simpson's-paradox problem, not real behavior. The new page's win was actually understated, since it got more low-intent users and still won.
Result
Recommended rebalancing samples with engineering and continuing the test, which prevented a wrong call from a broken setup.
% already-registered of roadblock clicks
New page
6.4%
Old page
31.7%
The imbalance that explained the "uneven funnel." Always check sample composition before trusting a headline lift.
The lesson worth drawing: a sample-ratio (SRM) check belongs in the pipeline, automated, before anyone reads the result.
Technical deep-dive: math, SQL & Pythonrepresentative
The math

A sample-ratio mismatch (SRM) check is a chi-square goodness-of-fit test against the intended split: χ² = Σ (observed − expected)² / expected. A tiny p-value means the randomization is broken. Then compare group composition (here, the share of already-registered users), because an imbalanced mix, not behavior, can move a step-level funnel rate (Simpson's paradox).

SQL: registered share by landing page
SELECT landing_page,
       COUNTIF(user_status = 'registered') AS reg,
       COUNTIF(user_status = 'anonymous')  AS anon,
       SAFE_DIVIDE(COUNTIF(user_status = 'registered'), COUNT(*)) AS pct_registered
FROM roadblock_clicks
GROUP BY landing_page;
-- new page: 6.4% registered   |   old page: 31.7% registered
Python: the SRM gate
from scipy.stats import chisquare
# observed exposures vs the intended 50/50
stat, p = chisquare(f_obs=[n_new, n_old], f_exp=[(n_new+n_old)/2]*2)
assert p > 0.001, "Sample-ratio mismatch; investigate before trusting the read"
Why it's advanced / how it's tested

The headline said "+18% CVR win"; the funnel said otherwise. Diagnosing it as a composition problem (registered users skip account creation), rather than accepting or rejecting the result, showed the new page was actually understated. The takeaway: an SRM check belongs in the pipeline as an automated gate, before anyone reads a result.

Recommender Test: Engagement vs. Subscriptions

Gannett · HP4U / Taboola · EXP-0035

The test
Taboola's personalized homepage vs. our standard homepage, ramped to 80% of audience.
The confounder we found
Top-line pages/session looked down, but auto-refresh pageviews were inflating the control. Removing them, the variant actually had higher real page depth.
How it was framed
An explicit tradeoff across ad revenue, CTR, story pageviews, and subscriptions, not a single up-or-down verdict.
Result
+48% homepage CTR, ~+8% downstream ad revenue (~$864K/yr) against a ~5% subscription dip. Leadership could weigh the tradeoff with eyes open.

Technical deep-dive: math, SQL & Pythonrepresentative
The math

The trap: total pageviews mix two populations: real reads and auto-refresh impressions. Decompose first: pv_total = pv_real + pv_autorefresh, then run the z-test on pv_real. Each KPI gets its own significance test; the decision is a multi-metric tradeoff, not one number.

SQL: strip the confounder before comparing
SELECT variant,
       SUM(pageviews)                               AS pv_total,
       SUM(IF(is_auto_refresh, pageviews, 0))       AS pv_autorefresh,
       SUM(IF(is_auto_refresh, 0, pageviews))       AS pv_real,
       SUM(ad_revenue)                              AS ad_rev,
       COUNT(DISTINCT IF(subscribed, user_id, NULL)) AS subs
FROM homepage_sessions
GROUP BY variant;
Python: per-metric significance + tradeoff
from statsmodels.stats.proportion import proportions_ztest
for metric in ['ctr', 'pv_real', 'subs']:
    z, p = proportions_ztest(num[metric], den[metric])
    flag = 'sig@95' if p < 0.05 else ('sig@90' if p < 0.10 else 'n.s.')
    print(metric, round(p, 4), flag)        # 90%-only flagged as elevated risk
ann_ad_rev = (ad_rev_var - ad_rev_ctrl) / weeks * 52   # translate lift to $/yr
Why it's advanced / how it's tested

Catching that auto-refresh inflated the control was the whole game; once removed, the variant's page depth flipped positive. we framed it as an explicit ad-revenue vs. subscription tradeoff (+~$864K/yr vs. ~5% sub dip) and held the line on significance: a related test that was only sig at 90% was flagged as elevated false-positive risk rather than calling it a win.

Causal Inference · Quasi-Experiment

Difference-in-Differences: BOGO vs. "2-For"

Molson Coors · Circle K FL · 22 test / 23 control stores

The question
Did "BOGO" promo language causally outsell "2-For," given we couldn't fully randomize, only observe selected stores?
What we did
Difference-in-differences: YoY % unit change for test vs. a predetermined control vs. a broad "general Florida" baseline, by segment and pack, with t-tests on the weekly differences.
The honest finding
Test differed from general trends, but not from the control, while control ≠ general. That means the control stores were poorly matched.
Result
Rather than declare a win, we recommended re-selecting better-matched controls before trusting the read. Integrity over a clean headline.
YoY unit change · test vs. control vs. general
TestControlGeneral FL
Test tracks the control almost exactly, so the promo shows no effect relative to control. Both diverge from "general," which exposed the control as mismatched.
Technical deep-dive: math, SQL & Pythonreal code
The math

Difference-in-differences: DiD = (Ȳ_test,post − Ȳ_test,pre) − (Ȳ_ctrl,post − Ȳ_ctrl,pre). It rests on the parallel-trends assumption: absent treatment, test and control would move together. It was operationalized as weekly year-over-year % change per store group, then t-tested the difference series.

Python: the three comparisons that exposed the bad control
from scipy import stats
# weekly year-over-year % change per store group
test_yoy    = [cy/ya - 1 for cy, ya in zip(test_cy,  test_ya)]
control_yoy = [cy/ya - 1 for cy, ya in zip(ctrl_cy,  ctrl_ya)]
general_yoy = [cy/ya - 1 for cy, ya in zip(gen_cy,   gen_ya)]

stats.ttest_ind(test_yoy,    control_yoy)   # vs designated control → not sig
stats.ttest_ind(test_yoy,    general_yoy)   # vs general FL         → significant
stats.ttest_ind(control_yoy, general_yoy)   # control vs general    → significant (!)
Why it's advanced / how it's tested

The honest read came from that third test: if the control already differs from the general baseline, it isn't a valid counterfactual, so we couldn't trust "test ≈ control = no effect." Rather than ship a null, we recommended re-selecting controls matched on pre-period sales trajectory (to satisfy parallel trends) before re-running.

Attribution & Survival

Two ways to value a decision

Gannett (MTA) · Empirical (survival curves)

Markov multi-touch attribution
Reconstructed user journeys from GPS-in-URL with 7/30/90-day lookbacks, then assigned channel credit by removal effect (how much conversion drops if a channel disappears) versus first/last/linear baselines. Shipped as a weekly Tableau dashboard.
Survival analysis (remediation)
Built survival curves for how fast CVEs are found and closed on client assets, then modeled the counterfactual if they prioritized via our model first.
The throughline
This work spans both bottom-up MTA and top-down MMM, and frames model value in business terms: faster remediation of the CVEs that actually matter.
Remediation survival curve (illustrative)
Today's prioritizationModel-prioritized
Share of important CVEs still open over time. Prioritizing with the model closes the high-risk ones faster, so the curve drops sooner.
Technical deep-dive: the Markov & survival mathrepresentative
Markov multi-touch attribution: the math

Model journeys as a Markov chain over channel states plus two absorbing states (conversion, null). The credit for channel c is its removal effect: 1 − P(convert | c removed) / P(convert | full graph), i.e. how much conversion probability collapses if that channel disappears.

import numpy as np
def p_convert(T):                       # absorption prob into 'conversion'
    Q, R = split_transient_absorbing(T) # Q: transient→transient, R: →absorbing
    N = np.linalg.inv(np.eye(len(Q)) - Q)   # fundamental matrix
    return (N @ R)[start_idx, conv_idx]

base = p_convert(T)
removal_effect = {c: 1 - p_convert(drop_channel(T, c)) / base for c in channels}
Survival analysis: the math

Kaplan-Meier estimate of "still un-remediated" over time: S(t) = Π (1 − dᵢ/nᵢ) for events up to t, with right-censoring for CVEs not yet closed. Compare the model-prioritized cohort's curve to the status-quo curve. The gap is faster remediation of the CVEs that matter.

from lifelines import KaplanMeierFitter
km = KaplanMeierFitter()
km.fit(durations=days_to_close, event_observed=is_closed)   # censored = still open
km.survival_function_                                       # S(t) curve
Why it's advanced

Absorbing-Markov math (fundamental matrix) for removal-effect credit beats heuristic first/last-touch; right-censored survival estimation is the correct tool for "time-to-remediation," not plain regression.

Technical deep-dive: math & Pythonreal code
The math

Silhouette for point i: s(i) = (b − a) / max(a, b), where a = mean intra-cluster distance and b = mean nearest-other-cluster distance. Pick the algorithm + parameters that maximize the mean silhouette, an objective answer to "how many clusters / which method," not a guess.

Python: scale → reduce → benchmark by silhouette
from sklearn import cluster, metrics
from sklearn.preprocessing import StandardScaler
from sklearn.decomposition import PCA

X = PCA(n_components=n).fit_transform(StandardScaler().fit_transform(df))

scores = []
for k in range(2, 12):
    labels = cluster.KMeans(n_clusters=k).fit_predict(X)
    scores.append((k, metrics.silhouette_score(X, labels)))
best_k = max(scores, key=lambda t: t[1])[0]

# ...repeated across KMeans, MiniBatch-KMeans, DBSCAN, HDBSCAN, Agglomerative,
#    Birch, Spectral, GMM over PCA *and* a Keras-autoencoder embedding.
Why it's advanced

Eight algorithms benchmarked across two representations (linear PCA and a nonlinear autoencoder), each tuned by silhouette, so the competitive-set clustering was chosen by evidence, not habit. The same discipline applies everywhere: the validation metric matches the problem.

Technical deep-dive: math & Pythonrepresentative
The math

Gradient-boosted trees fit an additive ensemble minimizing L = Σ logloss(yᵢ, ŷᵢ) + Σ Ω(treeₖ), each new tree pointed down the gradient of the loss. Positives are rare, so they get weighted (scale_pos_weight = neg/pos) and judge the model on PR-AUC + calibration, not accuracy (which a "predict-never-exploited" model would ace).

Python: training with imbalance + early stopping
import xgboost as xgb
params = dict(objective='binary:logistic', eval_metric='aucpr',
              max_depth=6, eta=0.05, subsample=0.8, colsample_bytree=0.8,
              scale_pos_weight=neg/pos)              # severe class imbalance
model = xgb.train(params, dtrain, num_boost_round=2000,
                  evals=[(dvalid, 'valid')], early_stopping_rounds=50)
Python: evaluate for rare positives
from sklearn.metrics import average_precision_score
p = model.predict(dtest)
average_precision_score(y_test, p)        # PR-AUC, the metric that matters here
# + isotonic/Platt calibration since the output is consumed as a probability
# + coverage/efficiency (gain) curves to show exploits captured per % reviewed

2,000+ features from 30+ sources, gradient boosting for nonlinear tabular interactions, imbalance handling, early stopping, and an evaluation stack built for rare events (PR-AUC, calibration, coverage/efficiency curves) rather than headline accuracy.

Data Engineering · Performance

Billion-Row Pipeline: 8h → 3.5h

Empirical Security · Spark → DuckDB · Databricks

The bottleneck
The training data is the hard part: intermediate sets balloon past 2 billion rows, and the original build took over 8 hours per run.
What we did
Audited the pipeline end to end, removed redundant steps, and shifted the heavy lifting from single-threaded Spark to multi-threaded DuckDB.
Kept it current
Added a delta process so the most recent month builds incrementally on a scheduled job, with no full rebuild.
Result
A full build (12 months, 300K+ vulnerabilities, 2,000 features) dropped to under 3.5 hours, and stays fresh automatically.
Full training-data build time
Original (Spark)
8.0 h
Rebuilt (DuckDB)
3.5 h
~2.3× faster. DuckDB's single-node, columnar, multi-threaded engine avoided Spark's cluster-shuffle overhead for this workload size.
Technical deep-dive: the engineeringrepresentative
The key insight

At this scale the workload is a big single-pass join/aggregate over columnar Parquet, not something that needs a cluster. Spark paid a shuffle tax and ran effectively single-threaded; DuckDB runs it multi-threaded on one node with no shuffle. Match the engine to the workload, then only rebuild what changed.

Python / SQL: multi-threaded DuckDB over S3 Parquet, incremental
import duckdb
con = duckdb.connect()
con.execute("SET threads TO 16")
con.execute("""
  CREATE TABLE model_input AS
  SELECT v.cve_id, f.*, l.label
  FROM read_parquet('s3://.../features/*.parquet')      f
  JOIN read_parquet('s3://.../vulns/*.parquet')         v USING (cve_id)
  LEFT JOIN read_parquet('s3://.../labels/*.parquet')   l USING (cve_id, as_of_date)
  WHERE as_of_date >= current_date - INTERVAL 30 DAY     -- delta: latest month only
""")
# 8 h  (Spark, distributed shuffles)  →  3.5 h  (DuckDB, single-node columnar)
Why it's advanced

The win wasn't a faster cluster; it was recognizing the workload didn't need one, removing redundant passes, and adding a delta build so the latest month is computed incrementally on a schedule. ~2.3× faster and self-refreshing. Same "profile, then pick the right tool" instinct as parallelizing the Molson ARIMA forecaster (multiprocessing vs. Dask).
